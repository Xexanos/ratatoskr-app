Ratatoskr · Android · v1

UX design for the Ratatoskr app

Ratatoskr is a remote control: play Audiobookshelf audiobooks on Sonos speakers, with progress synced back. The design turns that technical role into a stance — calm, warm, honest about state. Six screens, one active session, no distractions.

Foundation · docs/SPEC.md §12–13 Palette · Color.kt (copper / ash leaf) System · Material 3, Compose

Principles

Four rules every screen decision hangs on

P1

The server owns the truth

The app shows state, it doesn’t own it. Every screen therefore has a designed loading, empty and error state — “nothing here” is a normal state, not an accident. Positions advance locally between polls and correct themselves quietly.

P2

Thumb first

Each screen’s primary action sits in the bottom third: the connect button, the sign-in button, play/pause. All touch targets are at least 48 dp. The top of the screen is for reading, the bottom for acting.

P3

Warm and calm

Listening to audiobooks is an evening ritual, not a feed. 60/30/10: warm neutrals carry the surface, dark warm gray the text, and copper is reserved for exactly one primary action per screen. Ash green marks the secondary — status, chips, success.

P4

Make trust visible

Self-signed certificate, self-hosted server: the app explains what it verifies. The fingerprint is formatted to be readable, trusting is a deliberate act, and a certificate change is loud — never quietly dismissible.

User flow

Set up once — after that: library → speaker → listen

Connect and sign-in are one-time hurdles and disappear from the back stack afterwards. The library is home; from there it’s two taps to a playing audiobook.

The mini player (see Library) is the second path from the library to Now playing — the active session is reachable in one tap from anywhere, without its own route.

Design tokens

The brand supplies the palette, Material 3 the grid

All values come from ui/theme/Color.kt and the logo: copper (the squirrel) as primary, ash-leaf green (the Yggdrasil frame) as secondary, on warm neutrals. Dynamic color is deliberately off — the brand stays consistent.

Iconography

One icon family: Material Symbols (androidx.compose.material.icons, already used across the app). No emoji as UI glyphs — emoji render in the device’s own emoji font, so they differ across manufacturers and OS versions, can’t take the brand tint, and read as decoration to TalkBack. Material icons are bundled vectors: identical everywhere, tintable, and each carries a label.

Decisions

  1. 1The fingerprint is the centerpiece, not fine print: monospace, wrapped into groups of eight, on its own surface. That makes it genuinely comparable, line by line, against the server’s terminal output.
  2. 2Explain instead of alarm. “Self-signed” is the normal case for a home server. The copy states what the app does (pin the fingerprint, warn on change) — which builds more trust than a yellow exclamation mark.
  3. 3“Trust and continue” is the screen’s only copper action; Cancel is deliberately text-only. The hierarchy says: this is the one decision here.
  4. 4The logo appears only in the idle state. Once the certificate shows, the brand steps aside — that screen belongs to the facts.
Δ vs. current implementation
  • Fingerprint grouped instead of one long string; certificate data as a card with key-value rows.
  • Pinning explanation with shield icon added; a visible Cancel action (today only “Try again” in the error state).

Sign in

Signing in with context: where do these credentials go?

The credentials belong to Audiobookshelf; the recipient is the user’s own server. The screen shows both — and promises the outcome that matters (staying signed in), not implementation details.

Idle — a trust chip instead of an anonymous form

Decisions

  1. 1The server chip at the top (lock + host + “trusted”) links this step to the certificate decision from a moment ago — the user knows where the credentials go.
  2. 2Promise the outcome, not the mechanism. The helper says “You’ll stay signed in on this device” — true because the encrypted refresh token keeps the session alive across restarts (SPEC §5). The copy deliberately makes no claim about password storage, so a future “remember credentials for automatic re-login” option can be added without contradicting anything we advertised earlier.
  3. 3Password visibility toggle (eye) — long generated passwords are common with ABS, and typing them blind produces failed attempts.
  4. 4Errors appear inline below the fields (“Wrong username or password”) and the button stays enabled — no dialog, no lost input.
Δ vs. current implementation
  • Server context chip and stay-signed-in helper are new; password toggle is new.
  • The button moves from the end of the form into the thumb zone (pinned at the bottom).

Decisions

  1. 1A “Continue listening” shelf, vertical, above “All books”. Resuming a book is the app’s main job, so the two-to-three most-recently-listened books sit at the top as a vertical section — not a horizontal carousel, which would hide rows off-screen and break the one-handed top-to-bottom scan the rest of the screen relies on. German headers: “Weiterhören” and “Alle Bücher”.
  2. 2The shelf is a view onto the library, not a partition of it. Shelf books keep their alphabetical place in “All books” — no deduplication. Pulling them out would make the complete library look incomplete and shuffle it as progress changes. For the same reason there is no badge marking shelf books in the browse list: the shelf already surfaces them, and a badge would only repeat that.
  3. 3Search hides the shelf. Search is a query over the whole library; while it is active the shelf disappears and only results show. A “continue listening” section during search would compete with the matches and answer a question the user isn’t asking.
  4. 4One rule for the shelf slot: data > error row > nothing. If the server returns shelf items, they show; if the shelf load fails, a single inline tap-to-retry row holds the slot (never a blank gap, never a global error — “All books” below still loads); if there are no in-progress books, the slot collapses and “All books” moves up. A missing shelf is a reportable error, not a cosmetic absence.
  5. 5A mini player instead of a “Now playing” text button. It exists only while a session is active, shows book, speaker and position, offers pause directly, and opens the full Now-playing screen on tap. It feeds off the same getCurrentSession polling — no new route, no new module.
  6. 6Every row answers “where was I?”: a progress bar plus time remaining (“4 h 12 min left”). Raw percentages are a server number; time remaining is the user’s question. Shelf rows and browse rows are the same row — identical shape, so the shelf reads as a shortcut into the list, not a different thing.
  7. 7Covers come from the server’s cover proxy (contract v1.4.0), loaded under the app’s own auth and never fetched from Audiobookshelf directly. A title-initial placeholder tile fills the same slot while a cover loads or when a book has none — so a coverless book looks finished, not broken. One cover tile serves both shelf and browse rows.
  8. 8Search as a round M3 search bar directly under the app bar — searching is this screen’s main job next to resuming. Finished books show a green check instead of a full bar.
  9. 9Reload is pull-to-refresh, with retry everywhere a load can fail. Swipe down to reload (standard M3 indicator, not the knot loader — it is a short inline wait). A failed first load shows the error state with a “Try again” button; a failed refresh over an existing list keeps the list and offers retry in a snackbar action — retry is always the first offered action.
Δ vs. current implementation
  • A “Continue listening” shelf (most-recently-listened first) now sits above an “All books” header; the browse list keeps every book in place, and a failed shelf load shows an inline retry row instead of blanking the top of the screen.
  • The text-button row (“Now playing” / “Settings”) is replaced by an M3 top bar with a gear icon plus the mini player.
  • List rows gain cover art (server proxy, with the title-initial placeholder tile), progress bars and time remaining instead of a plain text label.

This is where users spend their time, often in low light — so the showcase is the dark variant (light follows the same rules). The transport layout follows the official Audiobookshelf app: chapter progress as the primary slider, book progress as a second thin bar, chapter skip on the outside.

Decisions

  1. 1“PLAYING ON Living Room” sits at the very top. For a remote, the most important information isn’t what’s playing but where — the audio comes from a device in the room, not from the phone.
  2. 2Chapter-first progress, like the official ABS app: the interactive slider covers the current chapter (elapsed left, remaining right), and a thin ash-green bar underneath shows the whole book. Copper = now, ash = the whole tree. Dragging the chapter slider is still one absolute seekSession under the hood.
  3. 3Five transport controls in ABS order: previous chapter · −10 s · play/pause (72 dp, copper) · +10 s · next chapter. The 10-second default mirrors ABS’s player default; the amount becomes a client setting later. Chapter skips are seeks to chapter marks — no new server capability.
  4. 4Stop is small, labeled and set apart: “Stop listening”. It ends the session; the confirmation that progress was saved arrives as a snackbar back in the library, so the button can speak the user’s language instead of the system’s.
  5. 5The status chip is the only motion (a softly pulsing dot while playing, static when paused/buffering). During a seek it shows “Buffering” until the server confirms. Position advances locally between polls and corrects quietly (SPEC §3).
Δ vs. current implementation
  • Speaker name in the header, chapter slider + book bar, and chapter skip buttons are new. Chapter navigation requires a SPEC §2 revision (currently listed as out of scope) and an additive contract change (chapter marks on item/session) — flagged in the implementation table.
  • ±10 s jumps are new (via existing seekSession); Stop moves out of the transport row into a labeled chip; the current single book slider becomes the thin secondary bar.

Empty search — the knot as a quiet brand moment

PEAK

Playback start

After the speaker tap the app switches to Now playing, the knot loader runs while the server connects to the speaker, the cover scales in gently, and the chip goes Buffering → Playing. No confetti — the real effect happens in the room. The app just confirms it calmly.

END

Stop with a receipt

“Stop listening” returns to the library; a snackbar says “Progress saved to Audiobookshelf” and the book’s row visibly carries the new bar position. Ending a session is a receipt, not a loss.

EMPTY

Emptiness as guidance

No speakers found → “Check that your Sonos speakers are on the same network as your server” plus a retry. Every empty state names whose state it mirrors (server, network, search) and what to do about it.

ERR

Errors stay local

Errors appear where the action was: inline under form fields, as a banner above lists, never as a global dialog. Retry is always the first offered action; an expired sign-in leads cleanly to the sign-in screen.

Motion & loading

The knot as the app’s pulse

For indeterminate full- and section-waits — the cold-start route decision, inspecting the certificate, connecting to a speaker, loading or searching the library, loading Now playing — the knot from the logo becomes the loading indicator: the real woven knot holds still while a copper squirrel runs its strand, trailing a fade along the band. It is held back behind a short time threshold, so only genuinely slow waits ever reach it. Ratatoskr, running the tree.

Live, embedded straight in the SVG (SMIL): the real woven knot sits underneath untouched, while a copper squirrel runs the strand’s own centreline trailing a comet fade. The generator emits this as docs/logo/ratatoskr-spinner.svg.

Rules

  1. 1The knot loader is every indeterminate full- or section-wait, but gated behind a time threshold. Rather than classifying screens by whether a wait feels slow — a line that shifts with the network, the Audiobookshelf instance and the device — every such wait renders nothing for ~500 ms and only then escalates to the knot. Fast waits never flash it; the (expensive) animation runs only in the genuinely slow case. Small inline affordances stay as the standard M3 circular indicator — the in-button sign-in spinner, pull-to-refresh, and the pagination footer while loading the next page — because a full knot doesn't belong inside a button or a list footer, and two spinners on one screen is one too many.
  2. 2The real knot, not an approximation. The background is the exact woven knot from the logo — nothing recoloured or cut out. On top, the squirrel runs that knot’s own strand centreline (the very curve the ribbon is buffered from, spikes and all), so it weaves over itself right where the logo does, instead of jumping between disconnected sub-shapes. One linear loop (≈ 4 s) reads as travel, not a blink.
  3. 3Reduced motion is respected: with animations off (read ANIMATOR_DURATION_SCALE, exposed via a LocalReducedMotion), the knot renders static at full opacity and the wait is communicated by text (“Connecting to Living Room…”). The demo on this page does the same via prefers-reduced-motion.
  4. 4A real comet, embedded in the SVG. A copper head at the squirrel ramps to transparent along ~a fifth of the strand behind it (arc length, so the fade hugs the band). It needs no JavaScript — the generator writes it as SMIL: a dash travelling the centreline plus the dot on an animateMotion path, so it animates as a plain SVG. Android’s VectorDrawable can’t run SMIL, so the app redraws the same in a Canvas: the centreline as a Path, a head offset on an infinite transition, the trail via PathMeasure.getSegment with an alpha ramp, the runner via getPosTan. No new dependency, no server involvement.
